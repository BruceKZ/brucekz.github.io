# In-Context Learning and Post-Training / 上下文学习与后训练

Covered in: Week 7 (`23-gpt-3+prompting+post-training`)

这一部分讨论的是大模型从“会续写文本”走向“能按用户意图工作”的过程。

## base model 是什么

`base model` 通常指只经过预训练的原始语言模型。

它已经学到：

- 文本分布
- 模式续写
- 大规模知识压缩

但它不一定擅长：

- 按指令回答
- 稳定遵守格式
- 拒绝危险请求
- 输出更接近用户偏好

`base model` 已经会“说语言”，但还不一定会“作为助手做事”。

这是课程后半段非常关键的区分，因为现代产品模型的大量可用性并不是在预训练那一步自动得到的。

## in-context learning

`in-context learning` 指的是：

- 不更新模型参数
- 只把任务说明、示例和输入放进上下文
- 模型就能在当前对话里完成任务

常见形式包括：

- `zero-shot`
- `one-shot`
- `few-shot`

这里关键的不是 shot 数量本身，而是：

- 模型不做参数更新
- 任务适配完全发生在上下文内部

这和传统 `fine-tuning` 是两套不同的任务适配思路。

## few-shot 的作用与局限

示例的作用不只是“给几个答案”，还包括：

- 说明输入形式
- 说明输出形式
- 暗示当前任务的映射关系

它的局限也很明确：

- 占用上下文长度
- 对示例顺序和措辞敏感
- 小模型通常更脆弱

这说明 `few-shot` 不是“给例子越多越好”的简单过程，而是一个受上下文预算和模型规模共同限制的机制。

## 为什么 in-context learning 会被视为范式变化

因为在更早的 NLP 思路里，一个新任务通常意味着：

- 重新训练
- 微调
- 单独做监督适配

而大模型出现后，很多任务可以先尝试：

- 写 prompt
- 给示例
- 直接在上下文里完成

这会极大改变任务开发和使用方式。

## emergent abilities

课程里把 `GPT-3` 之后的一些现象归纳为规模效应下更明显的能力，例如：

- 更强的 `few-shot learning`
- 更稳定的格式跟随
- 更好的跨任务泛化

这里的重点不是神秘化“涌现”，而是理解规模对任务归纳能力的影响。

更准确地说：

- 某些能力在小模型上很弱、不稳定或几乎看不见
- 规模扩大后，这些能力才开始变得明显且可用

课程要传达的重点是“规模改变能力曲线”，而不是把涌现理解成玄学事件。

## chain-of-thought prompting

课件里明确提到了 `chain-of-thought`。

它的基本想法是：

- 让模型显式输出中间推理步骤

在复杂推理任务上，这常常能提升表现，尤其是较大模型。

这背后的直觉是：

- 把推理拆成中间步骤
- 比直接跳到最终答案更容易得到稳定结果

它不保证一定正确，但常常能改善复杂推理题的表现。

## prompt search 与 prompt tuning

课件里不只讲了手写 prompt，还提到了：

- `AutoPrompt`
- `soft prompts / prompt tuning`

这说明 prompt 不只是文本技巧，也可以被搜索或参数化学习。

这说明“prompt engineering”不只是经验写作技巧，还可以进入更系统的优化框架。

从课程主线看，这一部分把“任务接口设计”也纳入了模型能力的一部分。

## instruction tuning

`instruction tuning` 的目标是让模型学会更自然地响应任务描述。

典型做法是：

- 收集大量 `instruction -> ideal answer` 数据
- 用监督学习继续训练模型

它可以看成在教模型“如何按任务说话”。

和预训练相比，它优化的不再是“语言本身最可能怎么延续”，而更像是在优化：

- 用户要你做什么
- 什么样的回答更符合任务要求

因此 instruction tuning 通常会显著提升：

- 指令跟随
- 格式稳定性
- 助手感

## instruction tuning 为什么不等于偏好优化

它解决的是：

- 模型是否会按任务要求回答

但还未必完全解决：

- 回答是否简洁
- 是否更有帮助
- 是否符合人类细腻偏好
- 安全边界是否合适

因此课程后面还要继续讲 reward model、RLHF、DPO。

## reward model 与 RLHF

做偏好优化时，常见流程是：

1. 收集人类偏好比较数据
2. 训练 `reward model`
3. 用强化学习继续更新策略模型

这套路线通常叫 `RLHF`。

这里的关键是把“人类更喜欢哪个回答”转换成模型能优化的训练信号。

`reward model` 的作用就是：

- 输入指令和候选回答
- 输出一个反映偏好质量的分数

有了这个分数，策略模型就能继续朝更高偏好方向优化。

## PPO

课件里明确提到了 `PPO`。

在这里需要知道的是：

- 它是 `RLHF` 里常见的策略优化算法
- 作用是利用奖励信号更新模型，同时尽量控制训练稳定性

课程里不要求把强化学习细节全部展开，但至少要知道：

- `PPO` 是 RLHF 流程中“怎么更新模型”的关键工具之一
- 它的存在说明后训练已经不只是普通监督学习

## 为什么 RLHF 不只是“再多训一点数据”

因为它改变的不是单纯下一个 token 的监督目标，而是：

- 用偏好信号引导整个回答风格

模型不只是学“什么是正确文本”，还学“什么是更受偏好的文本”。

## DPO

`DPO = Direct Preference Optimization`

它的基本思路是：

- 更直接地从偏好数据优化模型
- 不必完整走一遍传统 `RLHF` 流程

它通常被看作更简洁的偏好优化路线。

它的重要性在于说明：

- 偏好学习不只有“奖励模型 + RL”这一条路线
- 也可以更直接地利用偏好对进行优化

因此现代后训练方法不是单一路径，而是一整类方法族。

## RLVR 与 test-time scaling

课件后半部分还提到了：

- `RLVR`
- `GRPO`
- `test-time scaling`

这部分反映的是更近一步的观点：

- 如果奖励可以验证，就可以在更结构化的任务上继续强化模型
- 推理阶段增加更多计算和更多生成 token，也可能换来更强能力

这说明课程后半段已经开始把问题推进到更现代的视角：

- 不只是“训练时怎么学”
- 也包括“推理时怎么花更多计算换更强表现”

因此模型能力开始被理解为：

- 训练能力
- 提示能力
- 推理时计算预算

三者共同作用的结果。

## base model 与 instruct model

一个高频区分是：

- `base model`：更接近原始语言模型
- `instruct model`：经过后训练，更适合按人类意图工作

前者不一定更差，而是目标不同。

这两者最容易混淆的地方在于：

- base model 往往知识和续写能力很强
- 但 instruct model 在真实交互里通常更顺手

所以不能只用同一套标准去判断哪个“更好”，而要看任务目标是什么。

## 从课程主线看这一章的意义

前面的课程内容在不断增强模型的表示、建模和生成能力。

这一章进一步说明：

- 能力本身不等于可用性

一个模型可能已经：

- 会推理
- 会生成
- 会续写

但还不够：

- 稳定
- 听话
- 对齐人类偏好

因此后训练阶段才成为现代模型系统设计里不可缺的一部分。

## 需要记住的结论

- `in-context learning` 不更新参数，只靠上下文做任务适配。
- `instruction tuning` 用监督数据把模型从续写器推向助手模型。
- `RLHF` 依赖人类偏好数据、奖励模型和策略优化。
- `PPO`、`DPO`、`RLVR` 都属于后训练阶段的重要关键词。
- `base model` 与 `instruct model` 的用途不同，不能混为一谈。
- 现代大模型的可用性，往往是预训练、提示、后训练和推理时策略共同作用的结果。
